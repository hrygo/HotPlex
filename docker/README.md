## 🚀 The Late-Injection Architecture

To minimize the "Code -> Container" feedback loop, we decouple the stable language environments (SDKs) from the volatile application binary (`hotplexd`).

1.  **Static Base (`hotplex:base`)**: A common foundation (Debian Bookworm) including system tools and environment settings. Built once, cached forever.
2.  **Binary Provider (`hotplex:artifacts`)**: A lightning-fast Go builder that compiles ONLY the `hotplexd` binary.
3.  **Tech-Stack Variants (`hotplex:go`, `:node`, etc.)**: Inherit from `base` and install specific SDKs.
4.  **Late Injection**: The `hotplexd` binary is copied from the `artifacts` provider as the **final layer** of every variant.

**Benefit**: Modifying Go code only invalidates the final few megabytes of the image. Rebuilding a 2GB Java image after a code change takes **< 10 seconds**.

## 🛠️ Build & Usage

The easiest way to manage these images is via the root `Makefile`:

### Basic Commands
```bash
# Build the default Go stack
make docker-build-go

# Build a specific stack (e.g., node)
make docker-build-stack S=node

# Build everything
make docker-build-all
```

### Configuration
Update your `.env` file to select the active image:
```bash
HOTPLEX_IMAGE=hotplex:go
```
